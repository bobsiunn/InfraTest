# 면접 과제: Degraded Compute Node

## 배경

오늘 오전부터 반도체 설계팀 연구자들의 불편 신고가 잇따르고 있습니다.
당신은 해당 컴퓨트 노드를 담당하는 인프라 엔지니어로, 지금 막 JIRA 티켓을 받았습니다.

> "컴파일 작업이 평소보다 엄청 느려요. 작업이 멈춘 것 같기도 하고요."
> "패키지 설치하려고 했는데 한참 걸리다가 되긴 됐어요."

지금까지 파악된 상황은 아래와 같습니다.

---

## 상황

1. **스토리지**: `/opt/eda/scratch`는 반도체 설계 연구자들이 EDA 툴로
   회로 컴파일(synthesis, place & route 등)을 돌릴 때 생성되는 중간 산출물을
   읽고 쓰는 작업 디렉토리입니다.
   회로 컴파일은 수천~수만 개의 소규모 셀 파일을 반복적으로 생성·수정·삭제하는
   워크로드입니다.

   현재 여러 연구자의 컴파일 작업이 상시 가동 중이며, 최근 동시 사용자가
   증가하면서 해당 경로의 파일 I/O가 느려지고 일부 프로세스가 응답 없이
   멈추는 현상이 보고되고 있습니다.

   모니터링 상 아래와 같은 지표가 관측됩니다.

   - **load average**: `12.5` — 동시에 실행/대기 중인 프로세스 수의 1분 평균.
     16 vCPU 장비 기준 정상치는 `~4.0` 수준
   - **CPU 사용률**: 유저 프로세스 `3%`, 커널 `5%`, **I/O 대기 `89%`** —
     CPU 대부분이 연산이 아닌 스토리지 응답을 기다리는 상태
   - **디스크 지표** (`iostat -x 1`):

     ```
     Device   %util   w_await
     dm-3     103.9   102.3 ms
     ```

   - 아래 명령 기준 정상 노드는 1초 내 완료되나, 해당 노드는 수십 초 소요

   ```bash
   time dd if=/dev/zero of=/opt/eda/scratch/test bs=4k count=100 oflag=direct
   ```

2. **네트워크**: 연구자들이 EDA 툴 실행 전 라이선스 서버 연결을 확인하거나,
   내부 패키지 저장소에서 의존성을 받아오는 작업(예: `apt install`, `dnf install`, `pip install`,
   내부 git 저장소 fetch 등)이 시작 직후 수 초간 멈췄다가 진행되는 현상이
   반복되고 있습니다. 작업이 완전히 실패하지는 않아 간과되어 왔으나,
   빈도가 높아지면서 연구자들의 불편 신고가 이어지고 있습니다.

   모니터링 상 아래와 같은 지표가 관측됩니다.

   - 정상 노드의 `curl https://pypi.org` 응답: `~0.2s`
   - 해당 노드의 응답: `~5.3s` (처음 접근하는 도메인마다 매번 동일한 지연 발생)
   - IP를 직접 지정한 통신(`ping 8.8.8.8`)은 정상 응답
   - `dig` 기준 DNS 응답 지표:

     ```
     ;; Query time: 36 msec
     ;; SERVER: 127.0.0.1#53(127.0.0.1)

     real    0m5.041s
     ```
